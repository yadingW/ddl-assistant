# DDL Assistant V1.0：30 天验证计划与阶段记录

本文用于记录 DDL Assistant V1.0 发布后的使用反馈和产品决策。

观察周期暂定为：

- 开始：待填写
- 结束：待填写
- 当前状态：待填写

本文不作为最终“成功报告”，也不记录无法准确获得的日活、解析成功率和效率提升比例。所有结论尽量基于可确认的事实和使用行为。

## 1. 核心验证假设

30 天验证主要回答以下问题：

1. 连续文本记录是否比逐条创建 Todo 更容易长期坚持；
2. 自动提取日期并按日聚合，是否能够降低事项从视野中消失的风险；
3. 桌面同类事项是否减少回到旧工具；
4. 打开速度和启动等待是否影响持续使用；
5. 本地规则是否覆盖典型场景中的高频表达；
6. 待跟进功能是否具有持续价值；
7. 使用者是否理解并接受连续便签模式；
8. 回车投递是否属于具有共性的需求；
9. 当前规则解析是否已经出现足够证据，值得启动自然语言 AI POC。

## 2. 记录原则

每周记录分成三类：

- **事实**：实际发生、能够确认的使用情况和反馈；
- **判断**：根据事实形成的暂时解释；
- **决策**：本周选择做、继续观察、暂缓或不做的事项。

明确不记录：

- 无法准确统计的每日打开次数；
- 没有测试样本支撑的解析成功率；
- 无法证明的效率提升百分比；
- 无法核实的节省工时；
- 将单个用户意见直接写成普遍需求；
- 仅出于礼貌或鼓励的正向评价。

## 3. 阶段记录指标

| 指标 | 记录方式 |
| --- | --- |
| 使用天数 | 本周实际使用该工具的天数；未完整记录时如实注明 |
| 旧工具回流 | 同类事项重新写回旧工具、便签或表格的次数与原因 |
| 遗漏事件 | 是否出现能够确认的重要节点遗漏 |
| 解析修正 | 日期、时间、项目、待跟进对象或视图显示错误 |
| 高频场景 | 本周最常记录的工作内容类型 |
| 打开阻力 | 是否因启动慢、入口难找或等待加载而放弃使用 |
| 待跟进价值 | 是否实际使用待跟进、复制和完成记录 |
| 使用反馈 | 试用者的具体行为反馈，剔除礼貌性赞美 |
| 规则错题 | 无法识别或误识别的脱敏输入 |
| 产品决策 | 做、继续观察、暂缓、不做 |

## 4. 阶段观察示例

> 本节用于展示记录方式，不作为 30 天最终结论。

### 4.1 使用情况

**事实**

- 持续记录使用情况，按周补充实际天数；
- 观察同类事项是否回流到旧工具；
- 观察临时记录是否能顺利汇总到应用；
- 记录常用窗口形态和打开频率；
- 记录是否出现新的重要节点遗漏。

**判断**

- 连续便签与 DDL 视图的组合是否进入稳定使用流程；
- 临时记录回流是否主要与记录速度有关；
- 小窗口和独立入口是否属于持续使用的重要条件；
- 观察期结束前不据此宣称产品已经长期验证。

### 4.2 高频场景

可观察的高频场景包括：

- 记录多个项目的时间节点；
- 粘贴沟通记录中包含多个日期的内容；
- 查看今天、本周和未来需要处理的事项；
- 跟踪卡在他人处、需要后续跟进的事项；
- 保留跨周的长期计划；
- 勾选完成并查看处理时间。

### 4.3 旧工具回流

**事实**

- 记录桌面同类事项是否回到旧工具；
- 记录临时事项是否需要额外整理；
- 记录启动步骤是否构成摩擦；
- 观察 PWA 独立入口是否降低打开阻力。

**后续记录需要区分**

- 因移动场景回流；
- 因应用启动慢回流；
- 因产品缺少功能回流；
- 因事项本身不需要进入 DDL Assistant。

### 4.4 已修正的早期问题

第一周同时包含 V1.0 封版前后的集中修正。已处理的问题包括：

- 具体时间节点的识别和展示；
- 今日视图中的重复日期或状态文本；
- 完成时间记录与历史归档；
- 备份恢复与 PWA 使用入口。

这些属于版本完善过程，不能单独作为长期产品价值的证明。

### 4.5 当前遗留摩擦

**已确认**

- “复制今日待跟进”输出仍需要手工删除部分重复信息；
- 应用打开后，页面框架先出现，用户内容会稍后集中显示；
- 曾出现跨周初始化完成前输入内容，随后被归入上周的问题；
- 临时记录的汇总路径仍需持续观察。

**待继续观察**

- `//` 备注方式在多行长备注场景下是否会造成重复输入负担；
- 使用者是否普遍需要回车投递；
- 启动等待是否会实际导致放弃使用；
- 待跟进功能是否会成为稳定高频能力。

多行备注目前先记录为候选摩擦。只有在实际使用中反复出现，才考虑增加块级备注或其他语法。

### 4.6 使用反馈

**事实**

- 使用反馈应记录输入习惯、原文回看需求和规则学习成本；
- 回车投递相关反馈应作为独立实验方向收集；
- 使用反馈应优先记录具体行为，不记录可识别个人身份的信息。

**判断**

- 回车投递可能存在明确需求，但样本仍少；
- 需要判断问题来自输入交互，还是来自日期规则学习成本；
- 当前不应把单个反馈直接并入稳定主应用；
- 使用反馈应优先记录实际使用行为，不以“觉得不错”等鼓励性评价作为验证依据。

### 4.7 第一周产品决策

**做**

- 优化“复制今日待跟进”；
- 修复跨日、跨周初始化顺序；
- 制作独立的回车投递实验页；
- 开始建立规则解析“错题本”；
- 收集使用反馈。

**继续观察**

- 启动速度及其对持续使用的影响；
- 待跟进功能的长期使用频率；
- 使用者是否愿意持续使用；
- 连续便签与回车投递的适用人群；
- 多行备注是否真的造成高频摩擦；
- 是否需要启动自然语言解析 AI POC。

**暂缓**

- 将回车投递直接并入主应用；
- 移动端专用快速入口；
- 完整历史周检索；
- 账号和跨设备同步。

**不做**

- 项目看板；
- 优先级和标签；
- 任意多层任务结构；
- AI 自动发送消息或自动修改原始数据。
