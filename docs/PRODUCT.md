# 产品问题定义：连续工作记录与任务提醒之间的断层

## 1. 问题背景

工作便签本 / DDL Assistant 起源于一个真实的多项目工作场景。

在项目密集阶段，工作事项可能同时来自邮件、聊天消息、临时会议、口头安排和既有项目表。一封项目邮件中可能一次列出多个跨周、跨月节点；日常工作中又不断出现材料修改、进度确认、对外发送和内部催办等零碎事项。

其中部分节点具有明确的流程要求：需要按期完成；确实无法完成时，也应在节点前发送延期说明。节点一旦从视野中消失，可能留下可被追责的流程缺口，也可能错过活动、发布或用户招募等固定窗口，导致后续工作整体顺延。

问题出现在记录与提醒之间。微信和普通便签适合快速记录，但内容容易分散在信息流中；Excel 适合管理标准化项目节点，但依赖用户主动打开和持续维护；Todo 工具适合明确的单项任务，但整段工作安排需要逐条拆分和设置日期；项目管理工具适合团队协作和规范排期，但个人临时事项的录入和维护成本较高。

当用户需要不断巡检多个工具，才能确认今天、下周和更远时间分别要处理什么时，遗漏就容易发生。

DDL Assistant 的目标是：

> 在尽量不改变连续记录习惯、不增加大量结构化维护的前提下，把原始工作信息中的日期节点重新集中呈现出来，降低事项遗漏风险。

## 2. 替代方案与当前工作流的摩擦

| 方案 | 适合的场景 | 对当前需求的主要摩擦 |
| --- | --- | --- |
| 微信自聊 | 手机临时记录、随手转存信息 | 工作信息与普通消息混杂，需要主动回看 |
| 普通便签 | 快速、连续地记录文字 | 缺少日期提取和统一的时间视图 |
| Excel 或表格 | 标准化项目和批量节点管理 | 需要持续维护，并依赖用户记得主动打开 |
| 逐条创建型 Todo | 明确、独立的单项任务 | 整段安排需要拆分录入并分别设置日期 |
| 结构化项目管理工具 | 团队协作、复杂排期和流程管理 | 对个人碎片事项较重，需要维护更多字段和规则 |
| DDL Assistant | 连续记录并从原文派生提醒 | 当前仅适合个人、本地和桌面为主的使用场景 |

上述方案各有适用范围。DDL Assistant 服务于现有工作流已经产生明显摩擦、又难以长期坚持复杂任务系统的用户。

## 3. 核心产品理念

### 先记录，再整理，最后才是任务

工作信息刚出现时，用户未必能够立即判断优先级、完整步骤或最终负责人。若每次记录都要先创建标准任务、维护标签和填写字段，输入阻力会明显增加。

DDL Assistant 将原始文本作为事实来源：

- 用户可以直接输入或粘贴整段工作安排；
- 文本输入后自动保存并持续解析，无需额外点击“提交”；
- 左侧保留原始上下文和工作草稿；
- 右侧从原文中派生今日、本周和未来的 DDL 视图；
- 点击任务可以定位到对应原文，并直接修改日期或内容；
- 任务完成状态和完成时间单独保存，不会删除或改写原始记录。

本周随记用于频繁新增和修改的近期事项；长线备忘用于保存包含多个跨周、跨月节点的完整安排。两类原文都会进入同一套日期解析流程，无需再次手工搬运节点。

V1.0 中的日期可能表示正式截止日期、计划处理日期或重新关注的提醒日期。系统暂不强制区分三种语义，具体含义由原文说明，以保持输入方式轻量。

## 4. 目标用户

目标用户通常具有以下工作状态：

- 同时处理多个项目和时间节点；
- 需要持续跟进、协调或催办他人；
- 工作信息经常以聊天、邮件和临时安排的形式出现；
- 习惯使用自然语言、便签或微信进行碎片记录；
- 很少使用，或难以长期坚持维护完整项目管理工作流；
- 已经因信息分散、主动巡检不足或整理成本过高而出现遗漏风险。

典型角色可能包括营销项目运营、产品运营、商务或售前支持、承担较多协调职责的产品岗位，以及小团队中的多职能人员。

已经能够稳定使用现有 Todo、Excel 或项目系统，并且没有明显遗漏和维护负担的用户，通常没有切换需求。

## 5. 核心使用场景

- 直接粘贴一段包含多个日期的项目安排，自动生成对应节点；
- 记录领导、同事临时交代的零碎事项，并补充计划处理日期；
- 保存跨度数月的长线项目节点，在临近时自动进入提醒视图；
- 标记已经交给他人、但仍需继续催办的事项；
- 在桌面或分屏小窗口中持续查看今日、本周和未来任务；
- 在具体时间到达时接收本地通知；
- 完成后记录实际处理时间，并在写周报时回看原始记录和历史事项；
- 导出本地数据，以便备份和恢复。

## 6. V1.0 产品范围

当前版本已完成：

- 连续文本随记和自动本地保存；
- 本周随记与长线备忘；
- 相对日期、明确日期和具体时间识别；
- 日期域继承及项目上下文继承；
- 今日、本周、未来和已逾期视图；
- 待催事项；
- 完成状态、完成时间和历史归档；
- 点击任务定位原文；
- 具体时间本地通知；
- JSON 备份与恢复；
- PWA 安装、离线缓存和小窗口适配；
- 一键启动及人工测试清单。

V1.0 使用确定性的规则解析，优先验证核心工作流是否成立，同时控制成本、安全和稳定性风险。

未来版本可能使用 AI 理解更自由的自然语言、语音或文件内容，减少用户学习输入规则的成本。早期阶段可以采用人工确认，或只提示低置信度结果。原始内容和人工修改能力会继续保留。

## 7. 当前观察、已验证结果与后续验证

### 已观察到的结果

- 完成 PWA 安装并固定到桌面和任务栏后，产品已经进入每天的工作流；
- 在桌面工作场景下，原普通便签已不再承担同类记录；
- 时间节点已经集中到同一视图中，减少了在微信、便签和 Excel 之间反复查找；
- 当前常用日期与时间表达已基本无需人工修正；
- 手机临时事项仍可能先记录在微信，再复制到 DDL Assistant；
- 用户对当天及后续任务的整体情况比过去更熟悉。

### 当前已知摩擦

- 数据量增加后，应用打开和数据显示存在一定卡顿；
- 移动端仍缺少足够顺手的快速记录入口。

### 后续重点验证

- 待催功能是否会在真实工作中持续产生价值；
- 外部用户是否也存在相同的问题和使用动力；
- 回车投递、历史消息流等输入偏好属于普遍需求，还是个别用户习惯。

下载量、注册量、虚构的解析成功率和无法证明的节省工时，不作为当前阶段的主要成功指标。

## 8. 明确不适合的场景

DDL Assistant 当前不适合：

- 多人协作、任务分配、权限和审批管理；
- 复杂资源排期、甘特图和任务依赖计算；
- 跨设备实时同步；
- 需要页面关闭后仍持续运行的强提醒场景；
- 自动代表用户发送延期邮件或其他对外信息；
- 主要用于记录没有时间属性的长期想法和 Backlog；
- 替代企业级项目管理系统。

产品负责让节点重新进入用户视野。项目是否暂停、是否需要延期、是否应当对外沟通，仍由用户结合真实业务状态判断。涉及外部影响的动作，需要用户确认。
