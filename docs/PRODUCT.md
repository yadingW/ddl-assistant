# 产品问题定义：连续工作记录与任务提醒之间的断层

## 1. 问题背景

工作便签本 / DDL Assistant 面向需要持续记录和整理事项的通用场景。

在事项密集阶段，记录可能同时来自沟通记录、临时会议、口头安排和既有便签。一段记录中可能一次列出多个跨周、跨月节点；日常使用中也会不断出现方案修改、进度确认和后续跟进等零碎事项。

其中部分节点具有明确的时间要求：需要按期完成；确实无法完成时，也应提前更新后续安排。节点一旦从视野中消失，可能错过固定时间窗口，导致后续工作整体顺延。

问题出现在记录与提醒之间。即时通信工具和普通便签适合快速记录，但内容容易分散在信息流中；表格适合管理标准化节点，但依赖用户主动打开和持续维护；Todo 工具适合明确的单项任务，但整段工作安排需要逐条拆分和设置日期；项目管理工具适合团队协作和规范排期，但临时事项的录入和维护成本较高。

当用户需要不断巡检多个工具，才能确认今天、下周和更远时间分别要处理什么时，遗漏就容易发生。

DDL Assistant 的目标是：

> 在尽量不改变连续记录习惯、不增加大量结构化维护的前提下，把原始工作信息中的日期节点重新集中呈现出来，降低事项遗漏风险。

## 2. 替代方案与当前工作流的摩擦

| 方案 | 适合的场景 | 对当前需求的主要摩擦 |
| --- | --- | --- |
| 即时通信工具 | 临时记录、随手转存信息 | 工作信息与普通消息混杂，需要主动回看 |
| 普通便签 | 快速、连续地记录文字 | 缺少日期提取和统一的时间视图 |
| Excel 或表格 | 标准化项目和批量节点管理 | 需要持续维护，并依赖用户记得主动打开 |
| 逐条创建型 Todo | 明确、独立的单项任务 | 整段安排需要拆分录入并分别设置日期 |
| 结构化项目管理工具 | 团队协作、复杂排期和流程管理 | 对个人碎片事项较重，需要维护更多字段和规则 |
| DDL Assistant | 连续记录并从原文派生提醒 | 当前仅适合本地和桌面为主的使用场景 |

上述方案各有适用范围。DDL Assistant 服务于现有工作流已经产生明显摩擦、又难以长期坚持复杂任务系统的用户。

## 3. 核心产品理念

### 先记录，再整理，最后才是任务

工作信息刚出现时，用户未必能够立即判断优先级、完整步骤或最终负责人。若每次记录都要先创建标准任务、维护标签和填写字段，输入阻力会明显增加。

DDL Assistant 将原始文本作为事实来源：

- 用户可以直接输入或粘贴整段工作安排；
- 文本输入后自动保存并持续解析，无需额外点击“提交”；
- 左侧保留原始上下文和工作草稿；
- 右侧从原文中派生今日、本周和未来的 DDL 视图；
- 点击任务可以定位到对应原文，并直接修改日期或内容；
- 任务完成状态和标记完成时间戳单独保存，不会删除或改写原始记录。

本周随记用于频繁新增和修改的近期事项；长线事项备忘用于保存包含多个跨周、跨月节点的长期计划。两类原文都会进入同一套日期解析流程，无需再次手工搬运节点。

V1.0 中的日期可能表示正式截止日期、计划处理日期或重新关注的提醒日期。系统暂不强制区分三种语义，具体含义由原文说明，以保持输入方式轻量。

## 4. 目标用户

目标用户通常具有以下工作状态：

- 同时处理多个项目和时间节点；
- 需要持续跟进、协调他人或记录后续安排；
- 工作信息经常以沟通记录和临时安排的形式出现；
- 习惯使用自然语言或便签进行碎片记录；
- 很少使用，或难以长期坚持维护完整项目管理工作流；
- 已经因信息分散、主动巡检不足或整理成本过高而出现遗漏风险。

典型用户包括需要同时处理多个时间节点、承担协调职责或频繁整理碎片信息的知识工作者。

已经能够稳定使用现有 Todo、Excel 或项目系统，并且没有明显遗漏和维护负担的用户，通常没有切换需求。

## 5. 核心使用场景

- 直接粘贴一段包含多个日期的项目安排，自动生成对应节点；
- 记录临时安排和零碎事项，并补充计划处理日期；
- 保存跨度数月的长期计划，在临近时自动进入提醒视图；
- 标记已经交给他人、但仍需继续跟进的事项；
- 在桌面或分屏小窗口中持续查看今日、本周和未来任务；
- 在具体时间到达时接收本地通知；
- 完成后记录完成状态，保存标记完成时间戳，并在写周报时回看原始记录和历史事项；
- 导出本地数据，以便备份和恢复。

## 6. V1.0 产品范围

当前版本已完成：

- 连续文本随记和自动本地保存；
- 本周随记与长线事项备忘；
- 相对日期、明确日期和具体时间识别；
- 日期域继承及项目上下文继承；
- 今日、本周、未来和已逾期视图；
- 待跟进事项；
- 完成状态、标记完成时间戳和历史归档；
- 点击任务定位原文；
- 具体时间本地通知；
- JSON 备份与恢复；
- PWA 安装、离线缓存和小窗口适配；
- 一键启动及人工测试清单。

V1.0 使用确定性的规则解析，优先验证核心工作流是否成立，同时控制成本、安全和稳定性风险。

未来版本可能使用 AI 理解更自由的自然语言、语音或文件内容，减少用户学习输入规则的成本。早期阶段可以采用人工确认，或只提示低置信度结果。原始内容和人工修改能力会继续保留。

## 7. 当前观察、已验证结果与后续验证

### 已观察到的结果

- PWA 可作为独立入口固定到桌面或任务栏；
- 时间节点可集中在同一视图中查看；
- 常用日期与时间表达可由本地规则处理；
- 临时记录可在后续整理时汇总到 DDL Assistant；
- 用户可在今日、本周和未来视图中查看整体安排。

### 当前已知摩擦

- 数据量增加后，应用打开和数据显示存在一定卡顿；
- 移动端仍缺少足够顺手的快速记录入口。

### 后续重点验证

- 待跟进功能是否会持续产生价值；
- 使用者是否存在相同的问题和使用动力；
- 回车投递、历史消息流等输入偏好属于普遍需求，还是个别用户习惯。

下载量、注册量、虚构的解析成功率和无法证明的节省工时，不作为当前阶段的主要成功指标。

## 8. 明确不适合的场景

DDL Assistant 当前不适合：

- 多人协作、任务分配、权限和审批管理；
- 复杂资源排期、甘特图和任务依赖计算；
- 跨设备实时同步；
- 需要页面关闭后仍持续运行的强提醒场景；
- 自动代表用户发送消息或其他对外信息；
- 主要用于记录没有时间属性的长期想法和 Backlog；
- 替代企业级项目管理系统。

产品负责让节点重新进入用户视野。项目是否暂停、是否需要调整日期、是否应当沟通，仍由用户结合实际状态判断。涉及外部影响的动作，需要用户确认。
