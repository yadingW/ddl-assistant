# DDL Assistant 产品路线图

本文用于管理 DDL Assistant V1.0 之后的投入边界、验证顺序和技术演进。路线图采用“版本目标 + 进入条件”的方式，不把设想写成对用户的固定承诺。

项目遵循三项原则：

1. 先解决已经出现的真实问题；
2. 实验能力先与稳定主应用隔离；
3. 只有获得使用证据后，才增加长期维护成本。

## 1. V1.0：稳定基线

V1.0 已跑通以下核心链路：

> 连续文本记录 → 本地规则解析 → 派生提醒 → 完成与历史归档

已完成能力包括：

- 连续工作随记；
- 日期与具体时间提取；
- 相对日期转换和日期域继承；
- 项目上下文继承；
- `>` 待催事项；
- `//` 备注排除；
- 今日、本周、未来和历史视图；
- 任务完成状态与 `completedAt` 记录；
- 应用运行期间的本地到点提醒；
- JSON 数据导出、导入校验和失败回滚；
- PWA 安装、离线缓存和小窗口使用；
- V1.0 测试清单与正式 Release。

### 已经获得的验证

- 连续随记能够替代原有桌面便签；
- PWA 独立入口降低了打开阻力；
- 本地规则可以覆盖当前个人工作中的高频表达；
- 日期视图能够支持日常任务推进；
- 原文与任务状态分离后，记录上下文和任务处理可以同时保留。

### 当前已知问题

- “复制今日待催”仍需要手工删除重复信息；
- 跨周初始化存在低频但影响数据正确性的风险；
- 打开应用后，页面框架和用户内容之间存在等待时间；
- 外部用户对连续便签和回车投递的偏好尚未充分验证；
- 现有规则对熟悉产品的用户有效，外部用户可能存在学习成本。

## 2. 迭代节奏

计划每周投入约 6—7 小时：

- 两个工作日晚间，每次约 2—3 小时；
- 周末约 3 小时；
- 工作日零碎时间只处理小修正。

求职投递与产品迭代并行。是否收到招聘反馈不会决定项目是否继续推进，招聘反馈用于调整材料重点和功能优先级。

同一阶段最多推进：

- 一个实验方向；
- 两个主应用修正项；
- 一组外部反馈收集。

## 3. V1.1：回车投递实验与外部反馈

V1.1 的目标是验证外部用户的输入习惯，并修正当前主应用中投入产出比较高的问题。

### 3.1 回车投递实验

已经完成一版“便签模式 / 投递模式”切换的可交互原型。

下一步先制作独立可用实验页，不直接修改稳定主应用。实验页至少支持：

- 输入单条内容后按回车投递；
- 投递后输入框清空；
- 原文进入可查看的历史流；
- 日期和任务继续进入 DDL 视图；
- 本地保存；
- 原文查看和修改；
- 与 V1.0 数据相互隔离。

选择独立实验页的原因：

- 主应用已经稳定进入日常工作流；
- 可以独立观察投递模式的真实使用行为；
- 失败时无需处理两套交互在主应用中的兼容问题；
- 获得明确证据后，再决定是否加入模式切换。

### 3.2 外部反馈验证

优先邀请朋友和少量外部用户试用，重点记录：

- 是否能在无说明情况下理解输入方式；
- 连续便签和回车投递分别适合哪些场景；
- 投递后是否需要频繁回看或修改原文；
- 历史流应该如何查找；
- 日期规则是否构成主要上手障碍；
- 一周后是否仍在使用；
- 用户产生了哪些规则解析失败样本。

所有解析失败样本进入“规则错题本”，后续仅针对具有共性的高频问题补充规则，避免提前扩张规则字典。

### 3.3 当前主应用修正

按优先级推进：

1. 优化“复制今日待催”，减少重复信息和手工删改；
2. 修复跨日、跨周初始化顺序，避免用户输入后才切换日期域；
3. 对启动过程进行性能测量，再决定是否优化缓存重建和集中渲染；
4. 继续保证小窗口和分屏场景可用。

移动端专用入口暂缓。手机上的临时记录继续使用微信，当前投入优先用于桌面主流程和外部用户验证。

### V1.1 完成条件

满足以下条件后，再决定是否把投递模式并入主应用：

- 至少 3 位外部用户完成实际试用；
- 能说明两种输入方式各自适合的场景；
- 投递后的原文去向和修改方式得到基本验证；
- 日期提取、任务状态和历史查看可以稳定工作；
- 主应用的待催复制和跨周数据风险得到处理。

若证据不足，投递模式继续保留为实验页，不进入稳定主版本。

## 4. V1.2：自然语言任务解析 AI POC

### 4.1 为什么考虑 AI

V1 的本地规则对当前个人使用已经有效，但外部用户的输入可能包含：

- 多个动作和多个日期混在同一段文本中；
- “下周中”“月底前”等模糊时间；
- 项目、任务和待催对象依赖上下文判断；
- 会议记录或聊天内容中混有大量背景信息。

AI 的价值需要通过 POC 验证，重点观察它能否降低规则学习成本，而非单纯增加“大模型功能”。

### 4.2 POC 目标

第一版只验证一项能力：

> 用户输入一段自然语言，系统返回日期、项目、任务和待催对象等结构化候选结果。

暂不加入：

- 周报生成；
- 延期邮件生成；
- 自动发送消息；
- 账号与云同步；
- AI 自动改写原文；
- AI 直接写入主应用任务。

### 4.3 交互流程

采用“AI 解析 → 用户审阅 → 确认写入”的流程：

1. 用户提交原始文本；
2. AI 返回结构化候选结果；
3. 系统标记不确定字段；
4. 用户修改或确认；
5. 确认后再进入实验数据；
6. 原始文本始终保留。

第一版放在独立实验页，与 V1.0 主应用隔离。

### 4.4 候选技术路线

参考架构为：

```text
PWA 实验页
    ↓
Serverless 安全无状态接口
    ↓
Dify 工作流或等价编排层
    ↓
大模型结构化解析
    ↓
JSON 格式与字段校验
    ↓
前端审阅和修改
```

Dify 属于候选编排方案，POC 阶段根据配置成本、调试能力和调用费用决定是否采用。

### 4.5 安全与降级原则

- API Key 不写入前端代码；
- 仅发送完成解析所需的文本；
- 测试数据优先使用脱敏样本；
- AI 输出必须经过结构校验；
- 结果必须允许用户审阅和修改；
- 接口超时、失败或结果无效时，保留原文并提示用户；
- 主应用继续使用本地规则，不因 AI 故障失去核心能力；
- 未经确认，不自动发送消息或修改业务数据。

### 4.6 POC 启动条件

AI POC 是隔离实验，可以在以下准备完成后启动：

- 收集一批真实、脱敏的输入样本；
- 样本同时包含规则可处理和规则处理失败的情况；
- 已明确需要输出的 JSON 字段；
- 已确定 API Key 的服务端保护方式；
- 已设置可接受的试验调用预算。

POC 不要求先证明 AI 一定会进入主应用。它的任务就是获得证据。

### 4.7 接入主应用的条件

以下条件同时满足后，才考虑在主应用中增加 AI 能力：

- 高频、明确输入能够稳定解析；
- 复杂输入能够主动标记“不确定”；
- 至少 3 位外部用户认为自然语言输入比学习现有规则更省事；
- 审阅和确认流程不会抵消效率收益；
- JSON 输出结构稳定；
- 延迟、调用成本和数据安全边界可接受；
- AI 不可用时，本地规则仍可继续工作。

### 4.8 暂停条件

出现以下任一情况时暂停接入：

- 复杂自然语言需求仅来自极少数边缘场景；
- 用户的主要问题来自输入交互，而非语义解析；
- AI 结果需要频繁人工纠错；
- API 和云端维护成本超出个人项目承受范围；
- 无法形成可解释、可审阅的结果；
- 敏感工作内容缺少可接受的保护方案。

## 5. V2.0：根据证据决定整合方式

V2.0 不预设必须加入全部实验能力。可能的方向包括：

### 方向 A：连续便签 + 回车投递切换

适用于两类输入习惯均有稳定用户，并且维护两套交互的成本可控。

### 方向 B：连续便签保持主流程，AI 辅助复杂输入

适用于现有用户继续偏好便签，外部用户主要受到规则学习成本影响。

### 方向 C：回车投递结合 AI 解析

适用于外部用户更习惯对话式输入，AI 审阅流程也已验证。

### 方向 D：实验能力不进入主应用

当需求弱、使用频率低或长期维护成本过高时，实验页停止继续投入，V1 稳定流程继续保留。

## 6. 后续候选能力

在输入模式和 AI 解析得到验证后，再排序以下能力：

1. 根据完成任务生成周报草稿；
2. 优化待催内容生成；
3. 延期说明或沟通草稿；
4. 完整历史周检索；
5. 启动性能优化；
6. 安全的服务端 AI 调用层；
7. 跨设备迁移或同步。

其中，周报生成优先使用已完成任务和 `completedAt` 作为主要依据，原文只用于补充上下文。

## 7. 当前明确暂缓

近期不投入：

- 移动端专用快速入口；
- 账号与云同步；
- 团队协作、权限与审批；
- 项目看板；
- 优先级和标签；
- 任意多层任务结构；
- 关闭应用后的强提醒；
- AI 自动发消息；
- AI 自动修改业务数据。

## 8. 版本管理建议

版本号用于区分稳定主应用与实验能力：

- `1.0.x`：Bug 修复、文案调整、样式和兼容性微调；
- `1.1.x`：回车投递实验、外部反馈验证和非破坏性体验修正；
- `1.2.x`：自然语言解析 POC，可先作为独立实验页面发布；
- `2.0.0`：交互模型或底层架构发生明显变化，例如 AI 能力正式进入主应用。

独立实验页需要明确标注 `experimental`，不与稳定主版本的能力承诺混淆。

## 9. 路线图原则

每项能力进入主应用前，都需要回答：

1. 它解决了哪个已经出现的真实问题；
2. 当前方案为什么无法满足；
3. 新能力会增加哪些成本和风险；
4. 什么证据能够证明它值得长期维护；
5. 失败后能否安全退回稳定流程。

路线图的目标是持续提高产品价值，同时保护已经稳定运行的核心工作流。
